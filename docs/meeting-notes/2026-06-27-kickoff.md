## 4. Proposed Phase-0 design — confirm before coding
> **PROPOSED** — ratify or adjust together. This seeded [`../DESIGN.md`](../DESIGN.md), which is now the **authoritative** copy: if anything below differs, **DESIGN wins**.

**Containers (`docker-compose.yml`):**
- `web` — Flask, the **only** exposed service: host **8000** → container 5000. Calls `ai` + `db` over the internal Docker network.
- `ai` — Flask, **internal only** (`expose`, no host port). Loads the baked RF model; serves `POST /predict`.
- `db` — MongoDB, **internal only**. Named volume for persistence. `healthcheck` + `depends_on`.

**AI contract (`web` → `ai`):**
`POST /predict`
- request: `{ "features": { sleep_hours, resting_hr, hr_change, fatigue, soreness, weekly_freq, training_load, calorie_balance, bodyweight_trend } }` (the 9 proposal features)
- response: `{ "state": <one of a small, flexible category set — finalized in data exploration>, "proba": { … }, "recommendations": [ … ] }` (key is `state`, matching DESIGN §3 + the contract test)
- Model **baked into the image** (`joblib`, **pinned sklearn**); scale via `ai` replicas + gunicorn workers (multiprocessing only for *measured* CPU-heavy work).
- **Fault tolerance:** if `ai` is unreachable, `web` returns a graceful "readiness unavailable" — it must not crash.

**Data model (Mongo — proposal §9):**
- `users` { username, password_hash }
- `profiles` { user_id, age, gender, height, weight, goal, training_frequency }
- `analysis_history` { user_id, readiness, classification, calories, timestamp }  ← written from Phase 2
- (`programs` → Phase 2)

**Auth flow:** register (werkzeug hash) → login (verify, issue session/token) → auth-gate `/profile`, predict-via-web, dashboard. Validate/whitelist input types before any Mongo query (NoSQL-injection defense).

**Config:** commit `.env.example` (**never** `.env`); `DEBUG` / `TESTING` flags; all paths via env so tests run on any machine.
